# ADR-0005: Unfold Vault snapshot store (test scaffold)

## Context

This record exists **only for automated and manual testing** of ADR ingestion, graph generation, and multi-service diagrams. It adds a **second fictional** runtime component (alongside Unfold Beacon in ADR-0004) so tooling can validate additional nodes, edges, and naming without shipping code.

## Decision

**Fictional change (not implemented):** add **Unfold Vault**, an internal service that stores immutable snapshots of `adrs/**` and generated artifacts keyed by commit SHA. Consumers (including Unfold Lens in a hypothetical “time travel” mode) would request a snapshot by ref or SHA; Vault would serve read-only content from object storage.

**Unfold Vault (fictional)**
- Internal service
- API layer (e.g. Hono or Fastify) in front of S3-compatible storage
- Writes triggered only by successful Unfold Engine runs (or by a dedicated post-build hook), not by direct user uploads
- Does not replace Git as source of truth; it is an optional cache for faster historical views and demos

Relationships (fictional): **Unfold Engine** → writes `architecture.json` to the repo and notifies **Unfold Vault** to pin a snapshot; **Unfold Lens** may read **live** `architecture.json` from GitHub (unchanged) or **Vault** for pinned revisions; **Unfold Beacon** remains notification-only (ADR-0004).

## Consequences

- **Test-only:** no code, infra, or buckets are implied by this document
- If this were real: storage cost, retention policy, and access control become new concerns
- Safe to delete or replace when tests no longer need another sample “new service” ADR
